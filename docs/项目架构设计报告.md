## 1. 执行摘要

### 1.1 项目定位

本项目旨在开发一款**生产级的本地化桌面应用**，为政府机关、企事业单位提供公文格式智能检测与优化服务。

**核心价值**：
- ✅ 本地运行，数据不上传云端，保障公文安全
- ✅ 一键检测与修复，提升公文处理效率
- ✅ 插件化 AI 架构，支持多家大模型
- ✅ 规则配置驱动，易于扩展与维护

### 1.2 技术路线

**桌面端 + 本地核心引擎**架构：

```
用户 → Electron Desktop (React UI)
          ↓
    FastAPI 本地服务
          ↓
   Python Core Engine
     ↓        ↓
Document   Rule Engine
Engine        ↓
          AI Provider
```

**不采用**纯 Web 服务模式，用户下载 EXE 后即可独立运行。

## 4. 核心模块设计

### 4.1 Document Engine（文档引擎）

#### 4.1.1 设计原则

🚫 **禁止直接修改 Word 对象**  
✅ **必须通过中间文档模型**

**数据流**：
```
Word .docx
    ↓
Document Parser（解析器）
    ↓
DocumentModel（JSON 模型）
    ↓
Rule Engine / AI（检查与分析）
    ↓
Modifier（修改器）
    ↓
Document Generator（生成器）
    ↓
Word .docx
```

#### 4.1.2 核心数据模型

已实现 `backend/core/document/models.py`，基于 Pydantic：

**主要类**：
- `DocumentModel`：顶层文档模型
- `Paragraph`：段落（含文本、样式、格式）
- `Run`：连续文本片段（含字体、字号、加粗等）
- `Table`：表格
- `HeaderFooter`：页眉页脚
- `PageSetup`：页面设置（边距、纸张大小）
- `DocumentMetadata`：元数据

**特点**：
- 完全独立于 python-docx 对象
- 可序列化为 JSON
- 支持版本控制与对比

### 4.2 Rule Engine（规则引擎）

#### 4.2.1 规则配置驱动

所有公文规则存储在 `rules/official/*.yaml`，不在代码中硬编码。

**规则文件结构**：
```yaml
template_name: "通知"
document_type: notice

# 格式要求
title:
  font: "方正小标宋简体"
  size: 22pt
  align: center

body:
  font: "仿宋_GB2312"
  size: 16pt
  first_line_indent: 2em

# 检查规则
check_rules:
  - id: CHK-N001
    name: "标题字体检查"
    severity: P0
    field: title.font
    expected: "方正小标宋简体"
    message: "标题应使用方正小标宋简体"

# 修复规则
fix_rules:
  - id: FIX-N001
    ref_check: CHK-N001
    action: set_font
    target: title
    value: "方正小标宋简体"
```

#### 4.2.2 规则引擎流程

```
1. 用户上传 Word 文档
   ↓
2. Document Parser 解析为 DocumentModel
   ↓
3. Rule Loader 加载对应公文类型的 YAML 规则
   ↓
4. Rule Checker 执行检查规则
   ↓
5. 生成检查结果列表（问题描述 + 严重等级 + 修复建议）
   ↓
6. 用户选择"一键修复"
   ↓
7. Rule Fixer 应用修复规则，修改 DocumentModel
   ↓
8. Document Generator 生成新的 .docx
```

#### 4.2.3 实现任务

| 模块 | 文件 | 状态 | 功能 |
|------|------|------|------|
| 规则加载器 | `loader.py` | ✅ 已实现 | 加载 YAML 规则文件 |
| 规则执行引擎 | `engine.py` | 🟡 待实现 | 统筹检查与修复流程 |
| 格式检查器 | `checker.py` | 🟡 待实现 | 执行 check_rules |
| 自动修复器 | `fixer.py` | 🟡 待实现 | 执行 fix_rules |

### 4.3 AI Provider 架构

#### 4.3.1 插件化设计

**禁止写死某一家 AI 模型**，必须支持多家 Provider。

**架构**：
```
AIProvider（抽象基类）
    ↓
├── OpenAIProvider（GPT-4、GPT-4o）
├── DeepSeekProvider（DeepSeek）
├── ClaudeProvider（Claude 3.x）
├── GeminiProvider（Google Gemini）
├── QwenProvider（阿里通义千问）
├── ZhipuProvider（智谱 GLM）
├── OllamaProvider（本地模型）
└── CustomProvider（自定义 API）
```

#### 4.3.2 统一接口

已实现 `backend/ai/base.py`，定义抽象接口：

```python
class AIProvider(ABC):
    @abstractmethod
    async def analyze(self, document_text: str) -> AIAnalysisResult:
        """分析文档，返回结构化问题列表"""
    
    @abstractmethod
    async def proofread(self, text: str) -> list[dict]:
        """校对错别字、标点问题"""
    
    @abstractmethod
    async def rewrite(self, text: str, context: str = "") -> str:
        """提供表达优化建议"""
    
    async def test_connection(self) -> bool:
        """测试连接可用性"""
```

**AI 返回格式**（统一 JSON）：
```json
{
  "issues": [
    {
      "type": "typo",
      "position": "paragraph:3:10",
      "before": "的地得",
      "after": "的",
      "reason": "用词不当"
    }
  ]
}
```

#### 4.3.3 配置管理

用户在设置页面填写：
- Provider 类型（下拉选择）
- API Key（加密存储）
- Base URL（可选，支持代理）
- Model 名称

存储在 SQLite `ai_config` 表，API Key 通过 `cryptography` 加密。

### 4.4 校审中心（Check Center）

#### 4.4.1 功能定位

类似 IDE 的问题面板，展示文档中所有检测到的问题。

**问题等级**：
- **P0**：格式错误（字体、字号、对齐、间距）
- **P1**：错别字、标点错误
- **P2**：表达优化建议

### 8.2 Phase 2：Document Engine

**目标**：实现 Word 文档解析与生成

**任务**：
1. 实现 `parser.py`：将 .docx 解析为 DocumentModel
   - 段落文本、样式、格式
   - 表格
   - 页眉页脚
   - 页面设置
2. 实现 `generator.py`：将 DocumentModel 生成为 .docx
   - 保留原文档样式
   - 应用修改后的格式
3. 单元测试（使用测试样本文档）

**验收标准**：
- 上传 Word → 解析 → 生成 → 输出 Word，内容一致

**时间**：2-3 天

### 8.3 Phase 3：Rule Engine

**目标**：实现规则检查与自动修复

**任务**：
1. 实现 `engine.py`：规则执行引擎

4. 实现 `detector.py`：检测常见格式问题
5. 实现 `modifier.py`：批量应用修复

**验收标准**：
- 上传"通知"文档 → 检查 → 返回问题列表

**时间**：3-4 天

| 风险 | 影响 | 应对策略 |
