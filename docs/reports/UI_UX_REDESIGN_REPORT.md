## 一、当前 UI 问题诊断

### 1.1 视觉重心与空间利用

| 问题 | 表现 | 影响 |
|------|------|------|
| 视觉重心偏左上 | 所有页面内容区从左上角开始堆叠，右侧大面积留白 | 1920宽屏下约40%空间浪费 |
| 工作台为空壳 | 统计数据、最近文档全部硬编码，无API对接 | 用户打开软件看不到任何真实价值 |
| 单栏布局为主 | 所有页面都是单列堆叠Card | 信息密度低，需反复滚动 |
| 无效留白 | `space-y-8`、`py-16`等间距过大 | 内容被撑散，缺乏紧凑感 |

### 1.2 设计系统碎片化

| 问题 | 具体表现 | 涉及文件 |
|------|----------|----------|
| 灰色系冲突 | 暖色调设计定义了`--muted-foreground: #6B5644`，但20+处使用Tailwind冷灰`text-gray-500/600` | CheckCenter, Templates, TemplateRules, AISettings |
| 严重度颜色3套方案 | badge.tsx用`bg-p0`(未定义)，CheckCenter用`bg-red-100`，旧版用hex色 | badge.tsx, CheckCenter.tsx |
| 按钮变体错位 | `default`变体是棕色，但所有CTA手动覆写`bg-accent`(橙色) | 全局 |
| 模态框3种模式 | Dialog组件 / 手撸overlay+Card(50%透明) / 手撸overlay+Card(80%透明) | Rules, Templates |
| 原生select与Radix Select混用 | Templates/TemplateRules用`<select>`，其他用`<Select>` | 4个文件 |
| alert()/confirm()泛滥 | 11处使用原生阻塞对话框 | DocumentProcess, CheckCenter, Templates, Rules |
| 响应式间距不统一 | 6种不同的padding模式 | 8个页面 |

### 1.3 功能性缺陷

| 问题 | 严重度 |
|------|--------|
| 工作台全部硬编码数据，无API调用 | P0 |
| 最近文档列表有cursor-pointer但无onClick | P1 |
| 模板中心5个按钮堆叠，信息过载 | P1 |
| 无全局错误边界，API异常导致白屏 | P1 |
| 无面包屑导航，深页面只能点返回 | P2 |
| 无暗色模式 | P2 |
| 所有loading状态都是全屏spinner，无骨架屏 | P2 |

## 二、用户工作流分析

### 2.1 核心用户画像

**目标用户**：政府机关办公室文员、秘书科工作人员  
**核心任务**：按照 GB/T 9704 标准快速校审和优化公文格式  
**技术水平**：基础办公软件操作能力，不需要技术背景  

### 2.2 典型工作流

```
日常路径（高频）：
  打开软件 → 工作台看概览 → 上传公文 → 自动检查 → 查看问题 → 一键优化 → 下载

深度路径（中频）：
  打开软件 → 模板中心 → 选择公文类型模板 → 下载模板 → 按模板写公文 → 上传检查

配置路径（低频）：
  AI设置 → 配置大模型 → 测试连接 → 保存
  规则管理 → 查看/导入规则 → 调整规则
```

### 2.3 信息架构重新规划

**核心原则**：工作台是入口和总控台，文档处理是主工作流，其余为辅助功能。

```
一级（核心工作区）：
  ┌─ 工作台（首页/总控台）        ← 信息最密集，实时数据驱动
  └─ 文档处理（上传+检查+优化）    ← 核心工作流一站式完成

二级（辅助功能区）：
  ┌─ 校审中心（历史校审记录+统计）  ← 从属文档处理
  └─ 模板中心（模板库+规则管理）    ← 合并规则管理

三级（系统配置区）：
  └─ AI配置 + 关于
```

## 三、页面布局重构方案

### 3.1 布局体系升级

**从**：左侧导航 + 单内容区  
**到**：左侧导航 + 主内容区 + 可选右侧面板

```
┌────────┬──────────────────────────┬─────────────┐
│        │                          │  右侧面板    │
│  侧边栏 │      主内容区             │ (≥1440px    │
│  w-60   │      flex-1              │  自动展开)   │
│        │                          │  w-72       │
│        │                          │             │
└────────┴──────────────────────────┴─────────────┘
```

**断点策略**：
- `< 768px`：侧边栏折叠为图标，隐藏右面板
- `768-1024px`：侧边栏展开，隐藏右面板
- `1024-1440px`：侧边栏展开，隐藏右面板
- `≥ 1440px`：侧边栏展开，右面板自动显示

### 3.2 工作台重构方案（首页）

**目标**：即使无文档数据，页面也信息丰富、专业感强。

```
┌─────────────────────────────────────────────────────────┐
│ 顶部欢迎栏：「公文智能校审工作台」+ 日期 + 系统状态指示灯    │
├────────────────────┬────────────────────────────────────┤
│                    │                                    │
│  统计概览卡(4格)     │  快捷操作区                         │
│  ┌────┬────┐       │  ┌─────────────────────────────┐  │
│  │文档数│校审数│      │  │ [上传公文] [选择模板] [查看规则]│  │
│  ├────┼────┤       │  └─────────────────────────────┘  │
│  │修复数│合格率│      │                                    │
│  └────┴────┘       │  最近文档列表(真实数据)                │
│                    │  ┌─────────────────────────────┐  │
├────────────────────┤  │ 📄 关于XXX的通知.docx    已优化│  │
│                    │  │ 📄 XXX请示.docx         待检查│  │
│  最近校审记录        │  │ 📄 XXX报告.docx         已检查│  │
│  (最近5条)          │  └─────────────────────────────┘  │
│                    │                                    │
├────────────────────┴────────────────────────────────────┤
│ 底部状态栏：AI模型状态 | 规则版本 | 系统版本                  │
└─────────────────────────────────────────────────────────┘
```

**数据来源**：
- 统计数据 → `GET /api/documents/` 客户端聚合
- 最近文档 → `GET /api/documents/?limit=10`
- AI状态 → `GET /api/ai/config/{provider}`
- 规则信息 → `GET /api/rules/?source=all`

## 九、总结

本重构方案将产品从"文件处理工具"升级为"公文智能校审工作台"，核心改变：

1. **工作台**从空壳变为数据驱动的总控台
2. **布局**从单栏变为双栏/三栏，充分利用宽屏
3. **设计语言**从碎片化变为统一的暖色调政务风格
4. **交互**从原生弹窗变为内联反馈系统
5. **信息架构**从扁平导航变为分层分组导航

分P0-P3四个优先级，共13项任务，预估总工时约35小时。
